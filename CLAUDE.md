# Git — Atualização Automática

Após concluir qualquer conjunto de mudanças neste projeto, **sempre** faça commit e push para o repositório remoto:

```
git add <arquivos modificados>
git commit -m "<mensagem descritiva>"
git push origin main
```

- Use mensagens de commit em português, descritivas do que foi feito
- Adicione apenas os arquivos modificados (nunca `git add -A` sem revisar)
- Nunca force-push em `main`
- Sempre faça push ao final de cada tarefa concluída

# Versão do Sistema

A versão fica em `src/lib/version.ts` (constante `APP_VERSION`) e exibida no rodapé do sidebar.

**Regra:** a cada conjunto de mudanças commitado, incremente a versão seguindo semver:
- `patch` (x.x.**N**) — correções de bugs, ajustes visuais, pequenas melhorias
- `minor` (x.**N**.0) — funcionalidades novas ou mudanças relevantes de UX
- `major` (**N**.0.0) — mudanças arquiteturais grandes (raramente)

Atualize `src/lib/version.ts` junto com os demais arquivos no mesmo commit.

## Supabase
- `createClient()` — cliente com sessão do usuário (respeita RLS)
- `createAdminClient()` — usa `SUPABASE_SERVICE_ROLE_KEY`, bypassa RLS; use para operações admin
- Projeto ID: definido em `NEXT_PUBLIC_SUPABASE_URL` (ver `.env.local`)
- RLS `profiles`: usuário lê/edita apenas o próprio; admins leem todos (`profiles_select_admin`)
- Notificações: INSERT via `adminClient` (sem política de INSERT — service role bypassa RLS)
- **Relações N→1** (ex.: `tags(name)`, `courses(name)`, `lessons(title)` em joins): o PostgREST retorna **objeto** em runtime, mas o client sem tipos gerados infere **array**. Nunca acesse com `.map()`/`[0]` direto — use `toOne()` de `src/lib/supabase/relations.ts`, que normaliza as duas formas. (Regressão real na v1.50.1: Marketing quebrou com `t.tags.map is not a function` só para usuários com tags.)
- Triggers em `profiles` disparados via auth (GoTrue) rodam com `search_path` restrito — sempre qualifique objetos com `public.` dentro de funções de trigger (ver migração 030).
- **RLS bloqueia joins embutidos para linhas de outro dono, silenciosamente** (sem erro, só vem `null`/vazio): se a query do MEMBRO (client de sessão) faz embed tipo `profiles!fk(full_name)` para um `id` que não é o dele mesmo (ex.: nome do admin responsável por um chamado), a RLS de `profiles` (só lê o próprio perfil) barra o join e o campo volta vazio — mesmo a linha principal sendo do próprio membro. Real em `getMyFeedbackReports` (`src/app/actions/feedback.ts`): resolvido trocando para `adminClient` com filtro explícito `.eq('user_id', user.id)` (a checagem de posse fica no filtro, não na RLS).
- Pacotes `@tiptap/*` precisam ficar todos na MESMA versão exata (ver `package.json`) — as extensões fixam peer dependency `@tiptap/core` na versão exata, não em range; um caret (`^`) num único pacote causa conflito de peer dependency no `npm install` assim que o tiptap publica um patch novo.
- `<Select>` (`src/components/ui/select.tsx`) é baseado em `@base-ui/react/select`, não Radix. **`<SelectValue />` sozinho renderiza o valor bruto**, não o label do `<SelectItem>` selecionado — ele não escaneia os itens renderizados. Precisa de `<SelectValue>{(v) => label[v]}</SelectValue>` (função de mapeamento) ou do prop `items` no `<Select>` raiz. Sem isso, o trigger mostra o `value` cru (ex.: um UUID) em vez do texto — bug real visto no seletor de "Responsável" em `feedback-panel.tsx`.
- **`toast()` (sonner) chamado no primeiro `useEffect` de montagem de uma página pode ser perdido silenciosamente.** O `<Toaster/>` (root layout) só recebe eventos publicados DEPOIS que seu próprio efeito de `ToastState.subscribe()` rodar — e esse subscribe não faz replay de toasts anteriores. Como `{children}` vem antes de `<Toaster/>` no JSX do `src/app/layout.tsx`, um `toast()` disparado no efeito de mount de uma página corre — e às vezes perde — a corrida contra o subscribe do Toaster (mesmo commit, ordem de efeitos por posição na árvore). Sintoma: nenhum erro, o toast simplesmente não aparece, um clique/ação do usuário depois disparando outro toast funciona normal (não é o Toaster que está quebrado, só a chamada síncrona no mount). Correção: empurrar a chamada com `setTimeout(() => toast(...), 0)`. Caso real: toast de "desconectado por inatividade" em `src/components/auth/login-form.tsx` após redirect de `idle-logout-guard.tsx`.

## Papéis e autorização (admin / collaborator / member)
- `profiles.role` aceita `'admin' | 'member' | 'collaborator'` (migração `036`). **Colaborador** = pessoa interna que cria/edita conteúdo apenas da sua área, e estuda como membro comum no `/dashboard` (sem preview de rascunho).
- **Áreas de colaborador** são configuráveis pelo admin (tabela `collaborator_areas`: `name` + `capabilities text[]`), geridas em Admin → Membros → "Áreas de Colaborador" (`collaborator-areas-manager.tsx`). Cada colaborador pertence a UMA área (`profiles.collaborator_area_id`). **Decisão: NÃO reutilizar `tags`** — tags têm semântica de visibilidade de aluno; misturar criaria risco de apagar uma "tag-área" sem perceber que dava poder de escrita.
- **Capacidades** (enum fixo em `src/lib/capabilities.ts`): `courses` (cursos+módulos+aulas), `trainings`, `marketing`, `comercial`, `aereo`, `famtours`, `grupos`. Premiação (TamoJunto Winners), PodViajar e Corrida de Vendas gravam settings globais (`site_settings`, sem `owner_area_id` — não fazem sentido "por posse"). **Desde v1.74.0, colaborador com a capacidade `marketing` também edita as 3** (antes eram admin-only puro, sem nenhuma capacidade cobrindo). `src/app/actions/marketing-settings.ts` usa `requireCapability('marketing')` nos 3 (`saveTamojuntoWinners`/`savePodviajar`/`saveCorridaVendas`); os 3 managers (`tamojunto-winners-manager.tsx`/`podviajar-manager.tsx`/`corrida-vendas-manager.tsx`) recebem a prop `canEdit` (computada em `admin/marketing/page.tsx` como `isAdmin || ctx.capabilities.includes('marketing')`, repassada via `marketing-tabs.tsx`) em vez do antigo `isAdmin` — mesmo padrão `<fieldset disabled={!canEdit}>` já usado no resto do painel.
- **Dono do conteúdo**: `owner_area_id` (nullable, NULL = global/admin) em `courses`, `training_items`, `marketing_items`, `famtours`, `grupos`, `commercial_conditions`. `modules`/`lessons` herdam via `course_id`. **Reorder global (cursos/módulos/treinamentos) é admin-only** — reordenar um subconjunto bagunçaria os índices; reorder de aulas (escopado ao módulo) exige posse do curso pai.
- **Colaborador VÊ TUDO, só EDITA o que tem capacidade + posse** (v1.71.0 — antes era "só vê o que é seu"). `/admin/cursos` e `/admin/marketing` (as duas únicas telas de conteúdo) mostram TODOS os cursos/treinamentos/itens/famtours/grupos/condições de TODAS as áreas pra qualquer colaborador, inclusive abas sem nenhuma capacidade relacionada (Comercial, Aéreo, Grupos, Corrida de Vendas, Premiação, PodViajar aparecem sempre, em modo leitura). Abrir o detalhe de um curso/módulo/aula de outra área **não bloqueia mais** — abre normal, só os controles de mutação ficam escondidos/desabilitados. Cada tela calcula `canEdit`/`canCreate` (`ctx.role === 'admin' || (ctx.capabilities.includes(cap) && item.owner_area_id === ctx.areaId)`) e repassa como prop pros componentes — **isso é só UX**, a segurança real continua 100% nos guards de mutação (item abaixo), que não mudaram. Padrão de UI pro modo leitura: `<fieldset disabled={!canEdit}>` em volta de formulários inteiros (desabilita inputs/botões nativamente em cascata) + condicionais explícitas pra overlays de upload e pro toolbar do `RichTextEditor` (prop `editable`, não é controle de formulário nativo, fieldset não alcança). Em listas de item (famtours/grupos/etc, fora de um único form), cada botão de Editar/Excluir/Publicar fica atrás de `{item.canEdit && (...)}` explícito — o link de ABRIR o detalhe fica sempre visível (é a porta de entrada pro modo leitura).

Guards de página: `requireContentPage()` (novo — só exige admin/collaborator ativo, sem checar capacidade nem posse, usado como entrada em `cursos/page.tsx`/`marketing/page.tsx`) substituiu `requirePageCapability` nesse papel; `requireCoursePage`/`requireModulePage`/`requireLessonPage` agora só confirmam que a entidade existe (404-like), sem bloquear por posse. `requireAdminPage` (admin-only) e `requirePageCapability` (capacidade sem posse) continuam existindo para as demais telas. **As policies RLS continuam admin-only de propósito** — a mutação do colaborador passa pelo guard e usa `adminClient`; se alguma action esquecer o guard/adminClient, a RLS bloqueia silenciosamente (0 linhas, sem erro) — mesmo gotcha da nota de feedback acima.

## Logout automático por inatividade (só área do aluno)
- Escopo é **só `/dashboard`** (membro/colaborador estudando) — `/admin` não tem esse guard, de propósito (decisão do usuário).
- `src/components/ui/idle-logout-guard.tsx` (montado uma vez em `dashboard/layout.tsx`): timer único (`setTimeout`, não polling) resetado por eventos nativos (mouse/teclado/scroll) **ou** pelo evento customizado `window` `lv:activity` (`ACTIVITY_EVENT` em `src/lib/idle-timeout.ts`) — isso é o que permite qualquer parte da árvore sinalizar atividade sem prop-drilling/Context. Aviso (`AlertDialog`) 60s antes do limite de 10min; no timeout, `signOut()` client-side + `window.location.href` (reload completo, não `router.push`, pra matar qualquer canal Realtime em memória).
- **Vídeo tocando conta como atividade** mesmo sem interação manual: `study-video-player.tsx` escuta `onStateChange` do YouTube IFrame API e, enquanto `PLAYING`, dispara `lv:activity` a cada 25s — sem isso, um aluno assistindo aula sem mexer no mouse seria deslogado no meio do vídeo.
- Sincronização entre abas via `localStorage` (`ACTIVITY_STORAGE_KEY`) + evento `storage`: atividade numa aba reseta o timer das outras, pra uma aba ociosa não derrubar a sessão enquanto o usuário está ativo em outra.
- 10min/60s são constantes fixas em `idle-timeout.ts` (não configurável pelo admin, decisão do usuário) — reduzir temporariamente esses valores é o jeito de testar sem esperar 10 minutos reais.
- Toast "desconectado por inatividade" no login via `?reason=idle` (não confundir com `?error=inactive` do `proxy.ts`, que é conta desativada pelo admin — semântica diferente, e hoje nem é lido pelo `login-form.tsx`).

## Notificações
- Tabela `notifications`: `id, user_id, type, title, body, link, read_at, created_at`
- Após inserir notificações: chamar `revalidatePath('/dashboard', 'layout')` para invalidar cache do unread count
- Tipos usados (destinatário entre parênteses): `new_training`/`training_replay`/`training_live_expired` (membros do curso), `announcement` (todos os membros), `lesson_published`/`module_published` (membros com acesso ao curso), `task_submitted` (admins), `task_graded` (membro dono da submissão), `task_opened`/`task_closing_soon`/`task_closing_tomorrow` (membros matriculados, via cron `src/app/api/cron/task-reminders/route.ts`), `community_new_post`/`community_new_reply` (admins), `community_reply` (autor do post), `new_feedback` (admins, ao abrir chamado), `new_member_pending` (admins, cadastro aguardando aprovação), `feedback_update` (membro dono do chamado, quando admin muda status ou responde em `src/app/actions/feedback.ts`).
- **Bug real corrigido (v1.66.0): `notifyCourseMembers()` (`src/app/actions/notifications.ts`) nunca notificou ninguém.** A query fazia `.from('member_courses').select('user_id')`, mas a coluna real é `member_id` (migração `012`, e toda outra leitura de `member_courses` no projeto já usava `member_id` corretamente). Coluna inexistente → Postgrest retorna erro → `memberships` vem `undefined` → `if (!memberships?.length) return` sai calado, sem exception. Resultado: `lesson_published`/`module_published` nunca chegaram a ninguém desde que esses tipos existem — nem notificação in-app, nem (agora) e-mail. Corrigido para `select('member_id')`. **O mesmo bug ainda existe** em `src/app/api/cron/task-reminders/route.ts:69` (não corrigido — esse endpoint parece nem estar agendado na VPS hoje, ver nota de e-mail abaixo; ao mexer nos lembretes de tarefa, corrigir esse `user_id` → `member_id` também).
- **Som/toast em tempo real é escopo limitado, não geral**: nem todo tipo acima toca som — só os que têm um componente cliente dedicado assinando Realtime. Hoje existem dois: `src/components/admin/admin-notification-sound.tsx` (admin, tipos `new_feedback`/`new_member_pending`, montado uma vez em `src/app/(admin)/admin/layout.tsx`) e `src/components/members/feedback-notification-sound.tsx` (membro, só tipo `feedback_update`, montado em `src/app/(members)/dashboard/layout.tsx` **só quando `isTester`**, já que só testador abre chamado). O sino do membro (`notification-bell.tsx`) fora desses tipos é só visual — busca ao clicar, sem realtime/som. Ao adicionar um novo tipo que mereça som, decidir explicitamente se cria/estende um desses componentes; não veio de graça por estar na tabela `notifications`.
- **Realtime + polling (híbrido)**: `notifications` está na publication `supabase_realtime` (migração `033`). **Realtime sozinho não é confiável** (confirmado empiricamente: entrega intermitente do evento `postgres_changes`, mesmo com a tabela corretamente na publication e RLS ok) — por isso os dois componentes acima também fazem polling a cada 15s via `getRecentSoundNotifications` (`src/app/actions/notifications.ts`, genérico — recebe a lista de tipos que interessa a cada chamador) como garantia, deduplicando por `id` num `Set` em ref para não tocar duas vezes se ambas as vias pegarem o mesmo evento. Se precisar de realtime em outra tabela, mesma receita de publication: `ALTER PUBLICATION supabase_realtime ADD TABLE x` (idempotente com `DO $$ ... EXCEPTION WHEN duplicate_object`) — mas considere se vale a pena também ter um fallback de polling.
- **Notificações de feedback linkam pro chamado específico, não só pra lista genérica (v1.75.0)**: os 6 `link` de `src/app/actions/feedback.ts` (`new_feedback`, e os 3 pontos que disparam `feedback_update`: atribuição/status/resposta) incluem `?report=<id>` (e `&tab=minhas` quando o destinatário é o dono do chamado, não um admin). `FeedbackPanel` (admin) e `MyFeedbackList` (membro/colaborador) ganharam a prop `initialOpenId` — ao montar, já abrem o card daquele chamado (mesmo fora do filtro de status ativo, graças ao `|| r.id === openId` que já existia em `FeedbackPanel`) e rolam até ele (`scrollIntoView`, via `id="feedback-report-<id>"` em cada card). `admin/feedback/page.tsx` e `dashboard/feedback/page.tsx` só repassam `searchParams.report` como esse `initialOpenId`. `notification-bell.tsx` ganhou mais um `.replace('/dashboard/feedback', '/admin/feedback')` (mesmo padrão já usado pra `/dashboard/comunidade/`) — sem isso, colaborador clicando na notificação de dentro do `/admin` seria jogado pra `/dashboard/feedback`, repetindo o bug da v1.73.1.
- **Flakiness pré-existente, não corrigida**: em testes automatizados com cliques muito rápidos em sequência, o clique num item do sino às vezes não navega (fica na página atual) — parece uma corrida entre o `onClick={onClose}` do `NotificationItem` (fecha o painel) e a navegação do `next/link` subjacente. Acontece pra QUALQUER tipo de notificação (não é específico de feedback, já existia antes da v1.75.0) e não foi reproduzido em cliques com timing humano normal — navegação direta pela URL sempre funciona. Não corrigido por estar fora do escopo do que foi pedido; se afetar uso real, investigar `notification-bell.tsx`.

## E-mail transacional (Resend)
- `src/lib/email.ts`: wrapper `send()` sobre a lib `resend` (free tier, ~600 e-mails/mês). **No-opa silenciosamente** se `RESEND_API_KEY` não estiver setada (`resend = null`) — nenhum e-mail quebra o fluxo que chamou, erro de envio só vai pro `console.error`. Variáveis: `RESEND_API_KEY`, `NOTIFY_FROM_EMAIL` (remetente), `ADMIN_EMAIL` (destino dos e-mails só-admin) — todas em `.env.example`. **Confirmado em 2026-07-19: `RESEND_API_KEY` não está setada no `.env` da VPS** — hoje NENHUM e-mail deste sistema sai em produção (nem os que já existiam antes deste commit). Pra ativar: criar conta grátis em resend.com, gerar API key, colar no `.env` da VPS e reiniciar o PM2; **também verificar um domínio em resend.com/domains** — sem domínio verificado, o Resend só entrega pro e-mail da própria conta cadastrada, não pros membros de verdade.
- Funções e pontos de disparo (todas fire-and-forget, sem `await` bloqueando a action):
  - `emailAdminNewMemberPending` → `ADMIN_EMAIL`, ao registrar (`auth.ts` `register()`).
  - `emailWelcomeOnRegister` → o próprio cadastrado, mesmo ponto — mensagem de "está em análise", **não** confundir com `emailMemberApproved` (mensagem de "acesso liberado, pode logar", disparada só depois que um admin aprova em `members.ts`).
  - `emailMembersNewAnnouncement` → todos os membros ativos, ao publicar um comunicado (`announcements.ts`).
  - `emailAdminNewFeedback` → `ADMIN_EMAIL`, ao abrir um chamado de feedback.
  - `emailCourseContentPublished` → membros com acesso ao curso, dentro de `notifyCourseMembers()` (`notifications.ts`) — cobre aula E módulo publicados num único ponto (os 5 call sites em `lessons.ts`/`modules.ts` não precisam saber de e-mail).

## Presença ("online agora") e tempo na plataforma
- `profiles.last_seen_at` + `profiles.total_time_seconds` (migração `037`) — heartbeat client-side, **não** Supabase Realtime Presence (mesmo motivo do híbrido acima: transporte WebSocket já documentado como não confiável nesta infra).
- `src/components/ui/presence-heartbeat.tsx` grava a cada `PRESENCE_HEARTBEAT_MS` (60s, `src/lib/presence.ts`) via `recordHeartbeat()` (`src/app/actions/presence.ts`), que chama a function Postgres `record_presence_heartbeat` — **um único UPDATE atômico** que soma ao `total_time_seconds` só o intervalo desde o `last_seen_at` anterior, e só se esse intervalo for "contínuo" (≤180s, hardcoded na function — não confundir com a constante JS `PRESENCE_ONLINE_WINDOW_MS`, que é sobre o que conta como "online", não sobre continuidade de sessão). Evita contar como tempo de uso o hiato entre sessões (aba fechada, reaberta dias depois) e evita race condition de duas abas do mesmo usuário.
- Diferente do `IdleLogoutGuard` (só `/dashboard`, detecta atividade real pra decidir deslogar): `PresenceHeartbeat` roda em `/dashboard` **e** `/admin`, sem detectar atividade — só "aba aberta com sessão válida" já conta como online. São features independentes, montadas lado a lado nos dois layouts.
- Leitura no admin: `presenceSinceIso()` (janela de 3min = 3x heartbeat, tolera heartbeats perdidos) filtra quem está online. Card "Online agora" no dashboard (`admin-dashboard-shell.tsx`) quebra por role; bolinha verde na tabela de Membros (`members-table.tsx`). Ambas as telas fazem polling via `<DashboardAutoRefresh />` (`router.refresh()` a cada 30s) — dashboard admin e tabela de Membros eram 100% SSR sem nenhum polling antes disso.
- Tempo total (`formatDuration()`) aparece só em dois lugares, de propósito (decisão do usuário): aba Desempenho do próprio perfil (`profile-tabs.tsx`) e tela de detalhe do membro no admin (`admin/membros/[id]/page.tsx`, onde substituiu uma estimativa falsa `aulas_concluídas × 10min` que existia antes) — **não** é coluna/ranking na tabela geral de Membros.

## Upload de imagem de capa (padrão `uploadXCover`) — cuidado com foto de celular
- Padrão usado em `famtours-manager.tsx`/`trainings-manager.tsx` (e potencialmente outros managers com capa): o `<input type="file">` guarda o `File` cru no state; o upload/compressão (`toWebP` em `src/lib/image.ts`, via `sharp`) só acontece **depois**, dentro da server action (`uploadFamtourCover`/`uploadTrainingCover`), chamada como Server Action do Next.js. Isso significa que o **arquivo original (não comprimido)** é quem precisa caber no limite de `experimental.serverActions.bodySizeLimit` (`next.config.ts`, hoje `10mb` — era `5mb`, subido na v1.68.1).
- **Bug real (v1.68.1)**: uma foto de celular direto da câmera (facilmente 5–15MB) estourava esse limite, e como não havia `try/catch` em volta do `await uploadXCover(file)`, o erro do Next.js (corpo grande demais) não virava um toast — quebrava a página inteira (`This page couldn't load`) sem log útil no servidor. Corrigido com duas camadas: (1) guard no `handleFileChange` recusando arquivo `> 8MB` na hora da seleção, com toast explicando; (2) `try/catch` em volta do bloco inteiro do `startTransition` do submit, com toast de fallback genérico. **Ao criar um novo manager com upload de capa, replicar as duas camadas** — não é só copiar o padrão de `cover_url`/preview.

## Sincronização automática de chamados com o Notion (`src/lib/notion.ts`)

- **Desde v1.73.0**, toda mutação de um chamado de feedback (`src/app/actions/feedback.ts`) espelha automaticamente na base "Chamados" do Notion — não depende mais de eu ser acionado. Mecanismo independente do conector MCP: usa a API REST do Notion direto via `@notionhq/client`, autenticado com uma **Internal Integration** própria do workspace (token em `NOTION_API_KEY`, chamada "Universidade LV" no Notion) — precisa estar compartilhada com a página "Chamados" (ou a página-mãe "Universidade LV") via "..." → Conectores, senão a API retorna 404 mesmo com o token certo.
- Env vars opcionais (`NOTION_API_KEY`, `NOTION_CHAMADOS_DATA_SOURCE_ID`) — mesmo padrão de no-op silencioso do `RESEND_API_KEY`/`email.ts`: sem elas, `notion.ts` não lança erro, só não sincroniza. **Confirmar que ambas estão setadas no `.env` da VPS** (além do `.env.local` local) — sem isso, o sync fica no-op em produção mesmo funcionando local.
- **`feedback_reports.notion_page_id`** (migração `044`) guarda o id da página do Notion criada para cada chamado — é o que permite os eventos seguintes (atribuição/status/resposta) atualizarem a MESMA página em vez de criar uma duplicata. Os 4 pontos de mutação em `feedback.ts` (cada um já insere uma linha em `feedback_events`) chamam o Notion no mesmo lugar, sempre fire-and-forget (sem `await` bloqueante): `submitFeedback` cria a página e grava o `notion_page_id` de volta; `assignFeedback`/`updateFeedbackStatus`/`addFeedbackNote` anexam uma linha na Timeline e atualizam a propriedade correspondente (Responsável/Status).
- **Como a Timeline é atualizada sem manipular blocks do Notion**: em vez de achar o `block_id` da tabela nativa via API de blocks, `notionAppendTimelineRow` usa o endpoint `pages.updateMarkdown` (comando `update_content`, busca-e-substitui) procurando a única ocorrência de `</table>` na página e inserindo a linha nova antes dela — funciona porque cada chamado tem exatamente uma tabela Timeline na página.
- **Rótulos de Status do Notion ≠ rótulos do app**: o Select "Status" na base usa `Aberto/Em andamento/Resolvido`; o `STATUS_LABEL` interno do app (usado em toasts/UI) usa `Finalizado` em vez de `Resolvido` para o mesmo `status: 'resolved'` — `notion.ts` tem seu próprio mapa, não reaproveita o do app.
- **"Resolução" não é reescrita automaticamente** (decisão deliberada, não lacuna) — fica como placeholder estático criado junto com a página ("Em aberto, sem resolução ainda."). A Timeline (atualizada a cada evento) já é a fonte viva de verdade; editar aquele parágrafo de prosa via API exigiria achar o block certo por uma chamada extra, para um ganho pequeno.
- **Chamados criados antes da v1.73.0 foram sincronizados manualmente e depois tiveram o `notion_page_id` preenchido via backfill** (script único, não faz parte do fluxo de código) — qualquer chamado novo daqui pra frente já nasce com o `notion_page_id` certo desde a criação, sem precisar de backfill.

## Log de Atividades Administrativas (Admin → Relatórios → aba "Atividades")

- Tabela `admin_activity_log` (migração `043`) — log transversal (sem FK de uma entidade só, ao contrário de `feedback_events`): `actor_id` (nullable, `ON DELETE SET NULL`) + `actor_name`/`actor_role` denormalizados, `action` (`create|update|delete|toggle|upload|reorder`, vocabulário fechado por CHECK), `entity_type` (string livre, vocabulário fechado em código via `ACTIVITY_ENTITY_LABELS`), `entity_id` nullable sem FK (não dá pra referenciar ~20 tabelas de uma coluna só — é log, não dado transacional), `entity_label` denormalizado, `detail` texto livre. RLS só-admin-lê, sem policy de INSERT (só `adminClient`, mesmo padrão de `feedback_events`/`notifications`).
- **Helper central `src/lib/activity-log.ts`**: `logActivity(ctx: AdminContext, {action, entityType, entityLabel, entityId?, detail?})` — busca `full_name` do ator (1 query extra por chamada, sem cache entre invocações) e insere a linha via `adminClient`. **Sempre chamado sem `await` (fire-and-forget)** nos ~90 pontos de instrumentação, com `try/catch` interno como segunda camada — o log nunca pode quebrar nem atrasar a mutação principal. `diffFields(before, after, fieldLabels)` compara um objeto "antes" (buscado do banco pré-update) com o payload novo e devolve só os *nomes* dos campos que mudaram (não os valores antigo/novo — decisão do usuário: resumo, não diff completo, para manter viável em ~90 pontos).
- **Cobertura**: TODAS as ~90 funções de mutação (create/update/delete/toggle/upload/reorder) nos ~20 arquivos de `src/app/actions/*.ts` chamam `logActivity`. Padrão por tipo de ação: create → loga após o insert (com `entityId` se o insert já retornava `id`, senão `.select('id')` foi adicionado); update → busca "antes" (reaproveitando um select que já existisse por outro motivo quando possível, ex. `updateTrainingItem` já buscava `prev` pra decidir se notifica) e usa `diffFields`; upload → `detail` descreve o arquivo; toggle → `detail` descreve o novo estado (`'ativou'`/`'publicou'`/etc); delete → busca o label ANTES de excluir (senão a linha já não existe mais pra consultar).
- **Exceções ao padrão**: `updateSettings` (`settings.ts`, ~75 chaves em `site_settings`) loga genérico sem lista de campos — diff granular seria impraticável e pouco útil de ler. `reorder*` não faz diff de campo (`detail` opcional tipo "moveu para cima"). `feedback.ts` tem seu próprio `requireAdmin()` local (retorna só `{userId}`, não o `AdminContext` completo de `authz.ts`) — usa um helper interno `toAdminContext(userId)` pra montar um `AdminContext` mínimo só pra poder chamar `logActivity`; `addFeedbackNote` só loga quando quem respondeu é admin (resposta do próprio membro no seu chamado não é "atividade administrativa", já fica registrada em `feedback_events`).
- **Guards de autorização corrigidos como pré-requisito** (v1.70.0): `certificates.ts` (issueCertificate/approveCertificate/revokeCertificate/updateCertificateSettings/uploadCertificateTemplate/saveCertificateNamePosition) não tinha NENHUM guard antes — qualquer usuário autenticado conseguia emitir/revogar certificado; ganhou `requireAdmin()` + trocou pra `createAdminClient()`. `togglePinPost`/`toggleLockPost`/`hidePost`/`hideReply`/`deletePost` (`community.ts`) também ganharam `requireAdmin()` (antes só a RLS + a UI escondendo os botões protegiam). **`deleteReply` ficou de fora de propósito**: o próprio autor (membro comum) também pode excluir a própria resposta (`canDelete = isAdmin || isAuthor` na UI) — `requireAdmin()` ali quebraria esse fluxo. Consequência: exclusão de resposta de terceiro feita por admin não aparece no log de atividades nesta v1 (lacuna consciente, não bug).
- **UI**: `src/components/admin/atividades-log.tsx` (server component) — lista paginada (offset simples, `range()` + `count:'exact'`) com filtros por querystring (`?tab=atividades&ator=&entidade=&acao=&de=&ate=&page=`, `<form method="get">`, mesmo padrão de tabs por `?tab=` já usado em Relatórios). Herda a proteção admin-only da própria página (`requireAdminPage()` em `relatorios/page.tsx`) — colaborador nunca vê a aba, mas suas ações continuam sendo logadas (`actor_role: 'collaborator'`, badge "Colaborador" na listagem).
